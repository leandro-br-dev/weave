================================================================================
FORMATO DE ARQUIVO DE NATIVE AGENT
================================================================================

CADA NATIVE AGENT É UM ÚNICO ARQUIVO .MD COM A SEGUINTE ESTRUTURA:

1. YAML FRONTMATTER (obrigatório):
   ---
   name: [nome do agente]
   description: [descrição curta]
   model: [modelo preferido - ex: sonnet, haiku, opus]
   tools: [lista de ferramentas disponíveis]
   color: [cor para identificação visual]
   ---

2. CONTEÚDO DO AGENTE:
   - Propósito e definição
   - Workflow detalhado
   - Princípios chave de operação
   - Quando usar/não usar
   - Ferramentas disponíveis
   - Formato de saída esperado
   - Regras de otimização
   - Exemplos de uso
   - Instruções especiais
   - Tratamento de erros

================================================================================
COMO CRIAR NOVOS SUB-AGENTES
================================================================================

PASSO A PASSO:

1. Crie um novo arquivo .md no diretório apropriado
   Exemplo: dev/debugger.md

2. Adicione o YAML frontmatter no início do arquivo:
   ```yaml
   ---
   name: [Nome do Agente]
   description: [Descrição curta em uma linha]
   model: sonnet
   tools:
     - Liste as ferramentas disponíveis
   color: [blue/green/red/yellow/purple/orange]
   ---
   ```

3. Escreva o conteúdo do agente:
   - Propósito claro e específico
   - Workflow bem definido
   - Princípios de operação
   - Quando usar (critérios claros)
   - Quando NÃO usar (limitações)
   - Ferramentas disponíveis
   - Formato de saída esperado
   - Regras de otimização
   - Exemplos práticos
   - Instruções especiais
   - Tratamento de erros

4. Valide a estrutura:
   ✅ YAML frontmatter válido
   ✅ Campos obrigatórios preenchidos
   ✅ Descrição clara do propósito
   ✅ Workflow bem definido
   ✅ Exemplos práticos incluídos

5. Teste o agente:
   - Use em cenários reais
   - Verifique a qualidade da saída
   - Ajuste conforme necessário

================================================================================
PIPELINE DE DESENVOLVIMENTO
================================================================================

Task Request
    ↓
Plan Analyst (Haiku)
  → Codebase Analysis Report
    ↓
Plan Architect (Sonnet/Opus)
  → Architectural Specification
    ↓
Plan Planner (Sonnet)
  → JSON Execution Plan
    ↓
Backend Dev (Sonnet) / Frontend Dev (Sonnet)
  → Implement tasks from the plan
    ↓
Quality Engineer (Haiku)
  → Type check + Linter + Validation
    ↓
Git Expert (Haiku)
  → Branch → Commit → Push → PR
    ↓
staging-pr-handler (Sonnet)
  → staging-build-validator (Haiku)
  → staging-code-reviewer (Sonnet)
  → Decision: MERGE or REJECT
    ↓
Review & Merge

MÉTRICAS DE DESEMPENHO
================================================================================

ECONOMIA DE CUSTOS:
  Antes (agentes gerais):  Custo elevado por tarefa
  Depois (native agents):   Custo otimizado por tarefa
  Economia:                 Significativa com uso de Haiku para tarefas simples

EFICIÊNCIA:
  Tokens:          Otimizados por especialização
  Velocidade:      Mais rápido com agentes focados
  Precisão:        Alta (especializado)
  Manutenibilidade: Alta (arquivo único)

VANTAGENS DO FORMATO SINGLE-FILE:
  ✅ Simples de criar e manter
  ✅ Fácil de versionar
  ✅ Rápido de carregar
  ✅ Auto-contido
  ✅ Fácil de compartilhar
  ✅ Sem preconceitos tecnológicos (genérico e adaptável)
